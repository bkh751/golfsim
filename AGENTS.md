# AGENTS.md

## 1. 목적
이 저장소는 브라우저 기반 골프 시뮬레이터 `golfsim`을 개발한다.
모든 작업은 다음을 우선한다.

- 일관된 사용자 경험
- 신뢰 가능한 결과 표현
- 작은 범위의 안전한 변경

제품 방향은 `docs/product.md`를 따른다.

## 2. 우선순위
의사결정 우선순위는 아래 순서를 따른다.

1. 사용자의 직접 지시
2. 현재 작업의 Issue
3. `docs/product.md`
4. `AGENTS.md`
5. 기존 구현 관례

충돌 시 임의로 결정하지 않는다.

## 3. 역할
- 기획: 목표, 범위, 수용 기준을 정의한다.
- 디자인: 화면과 상호작용을 개발 가능한 형태로 정의한다.
- 개발: 승인된 범위만 구현하고, 테스트 결과와 리스크를 남긴다.
- 리뷰: 회귀, 범위 이탈, 검증 누락을 우선 확인한다.

## 4. 통신 채널
- `GitHub Issue`: 작업의 기준 문서. 목표, 범위, 수용 기준, 결정 사항을 기록한다.
- `GitHub Project`: 작업 상태를 관리한다. 상태는 `Backlog`, `Ready`, `In Progress`, `Blocked`, `Review`, `Done`만 사용한다.
- `AGENTS.md`: 협업 규칙의 기준 문서다.
- `docs/product.md`: 제품 방향의 기준 문서다.

## 5. GitHub Issue 운영
- 작업은 `대표 이슈 1개 + 팀별 하위 이슈` 구조로 관리한다.
- 기획, 디자인, 개발은 각자 자기 팀 이슈를 만든다.
- 모든 팀 이슈 본문 상단에는 `Parent: #<대표 이슈 번호>`를 명시한다.
- 대표 이슈는 목표, 전체 범위, 비범위, 최종 수용 기준, 연결 이슈를 가진다.
- 대표 이슈 본문에는 `Linked planning`, `Linked design`, `Linked development`를 기록한다.
- GitHub Project 상태는 대표 이슈만 기준으로 관리한다.
- 팀별 상세 내용은 해당 팀 이슈 본문이 기준 문서다.
- 범위 변경은 코멘트가 아니라 해당 이슈 본문 수정으로 반영한다.
- PM은 이슈 템플릿, 라벨, 트리아지, 대표 이슈와 하위 이슈 연결 품질을 관리한다.

## 6. 팀 간 규칙
- 기획/디자인 변경은 반드시 `GitHub Issue` 본문에 반영한다.
- 코멘트만으로 범위를 바꾸지 않는다.
- 개발은 요구사항이 불명확하거나 충돌이 있으면 즉시 Issue에 남긴다.
- 완료 보고에는 구현 내용, 테스트 결과, 남은 리스크만 남긴다.

## 6-1. Git 커밋 정책
- 구현 작업이 의미 있는 체크포인트에 도달하면 에이전트는 알아서 커밋해도 된다.
- 커밋은 항상 응집된 변경 단위로 나눈다. unrelated change를 묶지 않는다.
- 커밋 메시지는 기본적으로 한글로 쓴다.
- 커밋 메시지에는 가능한 한 현재 작업의 Issue 번호를 참조한다.
- 대표 형식은 `<한글 요약> (#<issue>)` 이고, 제목에 넣기 어렵다면 본문에 `Refs: #<issue>`를 남긴다.
- 커밋 전에는 관련 테스트를 통과시키거나, 테스트를 못 돌린 이유를 설명 가능해야 한다.
- 반복되는 커밋 판단과 메시지 작성에는 `skills/git-commit-team-ko` 스킬을 우선 사용한다.
- 사용자가 명시하지 않으면 `--amend`, reset, force push, 광범위한 add는 기본 동작으로 사용하지 않는다.

## 6-2. Codex 컨텍스트 정책
- Codex 세션 컨텍스트 사용량은 항상 15% 이하를 목표로 유지한다.
- 작업 시작 전에 필요한 파일과 문맥만 읽고, 넓은 범위의 일괄 읽기나 중복 탐색은 피한다.
- 긴 파일은 필요한 구간만 읽고, 이미 확인한 내용은 원문 재출력 대신 짧은 요약으로 재사용한다.
- 하위 에이전트도 같은 15% 규칙을 따른다. 즉시 필요한 질문만 위임하고, 동일 탐색을 중복으로 시키지 않는다.
- 큰 변경은 한 번에 전체 문맥을 싣지 말고, 파일 소유 경로와 핵심 상태 전이만 기준으로 잘게 나눠 진행한다.
- 응답에는 결론, 위험, 다음 단계만 남기고 장문의 로그/코드/출력 복붙은 금지한다.
- 컨텍스트가 15%를 넘길 가능성이 보이면 먼저 요약, 파일 범위 축소, 단계 분할 중 하나를 수행한 뒤 계속한다.

## 7. 완료 기준
아래를 모두 만족해야 완료다.

- 승인된 범위를 충족했다.
- 수용 기준을 충족했다.
- 관련 테스트를 통과했거나 누락 사유를 기록했다.
- 미해결 의사결정을 숨긴 채 완료 처리하지 않았다.

## 8. 대화 프로토콜
- 에이전트 간 통신은 사용자 중계를 통해서만 수행한다.
- 에이전트는 다른 에이전트와 직접 통신한다고 가정하지 않는다.
- 모든 응답은 아래 형식을 따른다.
  - 상태
  - 이해한 범위
  - 결과
  - blocker
  - 다음 요청
- blocker가 없으면 `blocker: 없음`으로 명시한다.
- 추가 입력이 필요하면 `다음 요청`에만 적는다.
- 완료 보고 시 `결과`에는 구현 내용, 테스트 결과, 남은 리스크만 포함한다.

## 9. 프롬프트 생성
- 팀 간 중계 프롬프트가 반복될 경우 `skills/relay-prompts` 스킬을 우선 사용한다.
- 이 스킬은 `AGENTS.md`, `docs/product.md`, 대표 이슈 정보를 기준으로 프롬프트를 생성한다.
- 스킬은 정책의 대체물이 아니라 반복 작업을 줄이기 위한 보조 도구다.
- 스킬이 생성한 프롬프트도 최종 기준은 항상 `AGENTS.md`와 현재 Issue 본문을 따른다.
