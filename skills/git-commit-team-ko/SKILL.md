# Git Commit Team (KO)

이 스킬은 이 저장소의 팀 공용 커밋 정책을 실행한다.

항상 먼저 확인할 문서:
- [`AGENTS.md`](/Users/user/workspace/game/golfsim/AGENTS.md)
- 현재 작업의 대표 이슈 또는 팀 이슈

## 기본 원칙

- 커밋은 "작업 단위가 설명 가능해진 시점"에 한다.
- 한 커밋에는 한 가지 응집된 변경만 담는다.
- 커밋 메시지는 기본적으로 한글로 쓴다.
- 커밋 메시지에는 가능한 한 이슈 번호를 참조한다.
- 이번 작업과 무관한 변경은 함께 커밋하지 않는다.

## 커밋 타이밍

아래 조건을 만족하면 적절한 타이밍으로 본다.

- 변경 목적이 한 문장으로 설명된다.
- 관련 테스트를 통과했거나, 테스트 미실행 사유를 설명할 수 있다.
- 바로 다음 작업이 완전히 다른 주제로 넘어간다.
- 중간 저장이 아니라 리뷰 가능한 체크포인트다.

아래 경우에는 커밋을 미룬다.

- 실험 코드만 있고 되돌릴 가능성이 높다.
- known broken 상태다.
- unrelated change가 섞여 아직 분리되지 않았다.

## 실행 워크플로우

1. `git status --short`로 변경 파일을 확인한다.
2. 필요하면 `git diff --stat`와 `git diff`로 범위를 검토한다.
3. 현재 작업의 이슈 번호를 확인한다.
   - 사용자 지시, 현재 이슈 문맥, 브랜치명, 최근 작업 기록 순으로 확인한다.
   - 번호를 확정할 수 없으면 임의로 만들지 않는다.
4. 이번 작업과 직접 관련된 파일만 스테이징한다.
5. 한글 커밋 메시지를 작성한다.
6. `git commit`을 수행한다.
7. `git show --stat --oneline -1`로 결과를 검증한다.

## 메시지 규칙

기본 제목 형식:

```text
<한글 요약> (#<issue>)
```

예시:

```text
결과 패널 가독성 개선 (#42)
백스핀 계산 오차 보정 (#108)
피팅 리포트 출력 형식 정리 (#131)
```

이슈 번호를 제목에 넣기 어려우면 본문 마지막 줄에 둔다.

```text
Refs: #42
```

메시지 작성 규칙:
- 제목은 짧고 구체적으로 쓴다.
- 구현 수단보다 변경 결과를 쓴다.
- 기본 언어는 한글이다.
- 저장소에 기존 prefix 관례가 강하면 prefix는 유지하되 요약은 한글로 쓴다.

## 안전 규칙

- 사용자가 명시하지 않으면 `git add .`를 기본값으로 쓰지 않는다.
- 사용자가 명시하지 않으면 `--amend`, rebase 수정, force push, reset은 하지 않는다.
- 충돌 상태이거나 훅 실패가 나면 원인을 요약하고 멈춘다.
- 이슈 번호가 필요한 저장소인데 번호를 끝내 확인하지 못하면, 커밋 전에 그 사실을 보고한다.

## 출력 형식

- `커밋 판단`
- `커밋 메시지`
- `이슈 참조`
- `포함 파일`
- `검증 결과`
